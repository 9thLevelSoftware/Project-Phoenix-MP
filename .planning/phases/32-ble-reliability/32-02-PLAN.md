<objective>
Harden the BLE state machine against four HIGH-severity issues: stale state observer corruption on reconnect (H2), unhandled onDeviceReady() failures (H3), missing scan timeout (H4), and the permission denied dead button on Android 11+ (H10).
</objective>

## Audit Findings

### H2: Stale peripheral state observer never cancelled on reconnect
- **File:** `shared/src/commonMain/kotlin/com/devil/phoenixproject/data/ble/KableBleConnectionManager.kt`
- **Lines 447-532:** In `connect()`, state observer launched with `peripheral?.state?.onEach { ... }?.launchIn(scope)`
- **Problem:** `cleanupExistingConnection()` sets `peripheral = null` but never cancels the old state observer Job
- **Impact:** Old observer fires stale `Disconnected` events → spurious `onReconnectionRequested()` → false disconnect state after successful reconnect

### H3: onDeviceReady() launched fire-and-forget
- **File:** `KableBleConnectionManager.kt`, line 469
- **Code:** `scope.launch { onDeviceReady() }` — no try/catch
- **`onDeviceReady()` does:** MTU negotiation, service enumeration, characteristic notification setup, polling start
- **Impact:** Kable exception during notification setup → crash with "Connected" already shown

### H4: startScanning() has no timeout
- **File:** `KableBleConnectionManager.kt`, lines 166-322
- **`SCAN_TIMEOUT_MS = 30000L`** exists in BleConstants but only used by `scanAndConnect()`, NOT by `startScanning()`
- **Impact:** If UI fails to call `stopScanning()`, scan runs forever draining battery

### H10: Permission denied loop on Android 11+
- **File:** `shared/src/androidMain/kotlin/com/devil/phoenixproject/presentation/components/BlePermissionHandler.android.kt`
- **Lines 204-268:** `BlePermissionDeniedScreen` shows "Try Again" button
- **Problem:** On Android 11+, after first denial, `ActivityCompat.shouldShowRequestPermissionRationale()` returns false — system silently ignores the launcher call
- **Impact:** "Try Again" button does nothing; user stuck on denied screen

## Key File Structure
- `KableBleConnectionManager.kt` (1107 lines): H2, H3, H4 changes all in this file
- `BlePermissionHandler.android.kt` (269 lines): H10 changes in this file
- `BleConstants.kt`: `SCAN_TIMEOUT_MS = 30000L` already defined, ready to use

## Task 1: Fix stale state observer (H2)

**Action:** In `KableBleConnectionManager.kt`, store the state observer Job reference and cancel it during cleanup.

**Specific changes:**
1. Add a property: `private var stateObserverJob: Job? = null`
2. In `connect()`, before launching the new state observer (~line 447):
   ```kotlin
   stateObserverJob?.cancel()
   stateObserverJob = peripheral?.state?.onEach { state -> ... }?.launchIn(scope)
   ```
3. In `cleanupExistingConnection()`, add at the top:
   ```kotlin
   stateObserverJob?.cancel()
   stateObserverJob = null
   ```
4. Ensure the state observer is also cancelled in `disconnect()` if it isn't already calling `cleanupExistingConnection()`

**Verify:** Read through the entire state observer flow to confirm no other code holds a reference to the old Job. Run build.

**Done when:** Each `connect()` call cancels the previous state observer before creating a new one.

## Task 2: Add error handling to onDeviceReady() (H3)

**Action:** In `KableBleConnectionManager.kt`, wrap the `onDeviceReady()` launch in proper error handling.

**Specific changes:**
1. Replace the fire-and-forget launch at ~line 469:
   ```kotlin
   scope.launch {
       try {
           onDeviceReady()
       } catch (e: CancellationException) {
           throw e // let cancellation propagate
       } catch (e: Exception) {
           log.e(e) { "onDeviceReady failed: ${e.message}" }
           logRepo.error(LogEventType.ERROR, "Device initialization failed",
               connectedDeviceName, connectedDeviceAddress, e.message)
           // Clean up the broken connection
           cleanupExistingConnection()
           reportConnectionState(ConnectionState.Disconnected)
       }
   }
   ```
2. Verify that `reportConnectionState(Disconnected)` is the correct terminal state after init failure (vs. an Error state)
3. Verify `cleanupExistingConnection()` properly nulls the peripheral and stops polling

**Verify:** Code review that all exception paths from `onDeviceReady()` (MTU, service discovery, notification setup) are now caught. Run build.

**Done when:** `onDeviceReady()` failures result in clean disconnection rather than crash.

## Task 3: Add scan timeout to startScanning() (H4)

**Action:** In `KableBleConnectionManager.kt`, wrap the `startScanning()` scan flow in a timeout.

**Specific changes:**
1. In `startScanning()` (~line 166), wrap the Scanner collection in `withTimeoutOrNull`:
   ```kotlin
   scanJob = scope.launch {
       try {
           withTimeoutOrNull(BleConstants.Timing.SCAN_TIMEOUT_MS) {
               Scanner { ... }.advertisements
                   .onEach { advertisement -> ... }
                   .catch { e -> ... }
                   .collect()
           }
           // Timeout reached — auto-stop
           if (/* still in scanning state */) {
               log.w { "Scan timeout reached after ${BleConstants.Timing.SCAN_TIMEOUT_MS}ms" }
               stopScanning()
           }
       } catch (e: CancellationException) {
           throw e
       } catch (e: Exception) {
           log.e(e) { "Scan error: ${e.message}" }
           reportConnectionState(ConnectionState.Disconnected)
       }
   }
   ```
2. Verify `stopScanning()` handles being called from within the scan coroutine (it cancels `scanJob`, which would be self-cancellation — may need to check)
3. If self-cancellation is an issue, instead of calling `stopScanning()`, directly update state and report

**Verify:** The scan timeout fires after 30s. Run build.

**Done when:** `startScanning()` auto-stops after `SCAN_TIMEOUT_MS` and transitions to Disconnected state.

## Task 4: Fix permission denied dead button (H10)

**Action:** In `BlePermissionHandler.android.kt`, detect permanent denial and show "Open Settings" instead of "Try Again".

**Specific changes:**
1. In `BlePermissionDeniedScreen`, get a reference to the current Activity (via `LocalContext.current as Activity`)
2. Check `shouldShowRequestPermissionRationale` for the required permissions:
   ```kotlin
   val activity = LocalContext.current as Activity
   val canRetry = BlePermissions.getRequiredPermissions().any { permission ->
       ActivityCompat.shouldShowRequestPermissionRationale(activity, permission)
   }
   ```
3. If `canRetry` is true: show "Try Again" button (existing behavior)
4. If `canRetry` is false: show "Open Settings" button that launches:
   ```kotlin
   val intent = Intent(Settings.ACTION_APPLICATION_DETAILS_SETTINGS).apply {
       data = Uri.fromParts("package", context.packageName, null)
   }
   context.startActivity(intent)
   ```
5. Add a `LaunchedEffect` that re-checks permissions when the Composable re-enters the composition (user returns from Settings):
   ```kotlin
   val lifecycleOwner = LocalLifecycleOwner.current
   DisposableEffect(lifecycleOwner) {
       val observer = LifecycleEventObserver { _, event ->
           if (event == Lifecycle.Event.ON_RESUME) {
               if (BlePermissions.arePermissionsGranted(context)) {
                   permissionState = BlePermissionState.Granted
               }
           }
       }
       lifecycleOwner.lifecycle.addObserver(observer)
       onDispose { lifecycleOwner.lifecycle.removeObserver(observer) }
   }
   ```

**Verify:** Build succeeds. Code review confirms the Settings intent uses correct action and URI scheme.

**Done when:** Permanently denied permissions show "Open Settings" button that opens the app settings page, and returning from Settings re-checks permissions.
