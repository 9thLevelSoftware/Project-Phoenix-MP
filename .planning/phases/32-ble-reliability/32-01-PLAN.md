## Task 2: Wire auto-reconnect consumer

**Action:** In `BleConnectionManager.kt`, add a collector in the `init` block that subscribes to `bleRepository.reconnectionRequested` and triggers a reconnection attempt.

**Specific changes:**
1. In the `init` block of `BleConnectionManager`, add a new collector:
```kotlin
scope.launch {
    bleRepository.reconnectionRequested.collect { request ->
        Logger.w { "Auto-reconnect requested: ${request.deviceName}" }
        // Don't reconnect if user explicitly disconnected or workout not active
        if (workoutStateProvider.currentWorkoutState.value is WorkoutState.Active) {
            delay(1500L) // BLE stack cooldown
            ensureConnection(
                onConnected = { Logger.i { "Auto-reconnect succeeded" } },
                onFailed = { Logger.w { "Auto-reconnect failed" } }
            )
        }
    }
}
```
2. Ensure the reconnect only fires during active workouts (use `workoutStateProvider` which is already injected)
3. The 1500ms delay is critical — Android BLE stack needs cooldown between disconnection and reconnection

**Verify:** The reconnection flow compiles. Verify via code review that `KableBleConnectionManager` correctly emits `onReconnectionRequested()` only when `wasEverConnected && !isExplicitDisconnect`. Run unit tests.

**Done when:** The `reconnectionRequested` flow has exactly one consumer in production code, and it triggers reconnection only during active workouts.
