## Audit Findings

### M1: Init collectors have no .catch{}
- **File:** `shared/src/commonMain/kotlin/com/devil/phoenixproject/presentation/manager/ActiveSessionEngine.kt` (3265 lines)
- **Init block collectors (lines 170-297):** 8 collectors launched in parallel
- **Specifically, collectors without .catch{} or try/catch:**
  - Collector #2 (~line 145): metricsFlow → `handleMonitorMetric()` — high-frequency, any NPE kills it permanently
  - Collector #5 (~line 224): deloadOccurredEvents → auto-stop logic
  - Collector #7 (~line 271): global metricsFlow (CRITICAL — matches parent repo pattern)
  - Collector #8 (~line 280): heuristicData → Echo mode force telemetry
- **Error handling pattern:** Guard clauses check `WorkoutState.Active` before processing, but no exception catching
- **Impact:** A single bug in `handleMonitorMetric()` kills the metrics collector permanently for the session — no more position/load updates reach the UI

### M2: CONNECTION_RETRY_DELAY_MS = 100ms
- **File:** `shared/src/commonMain/kotlin/com/devil/phoenixproject/util/BleConstants.kt`, line 163
- **Current value:** `100L` (100 milliseconds)
- **Android BLE requirement:** Minimum 1-2 seconds between GATT connection attempts to the same device
- **Impact:** Rapid retries hit the still-busy GATT stack, guaranteeing all retries fail; can leave zombie GATT connections on Android 12+

## Key Architecture Notes
- ActiveSessionEngine collectors run in `scope` (viewModelScope via WorkoutCoordinator)
- `scope` uses SupervisorJob — uncaught exceptions from `launch {}` propagate to uncaught exception handler
- On Android, uncaught coroutine exceptions crash the app
- Collectors #6 (`repEventsCollectionJob`) and #7 (`monitorDataCollectionJob`) have named Job refs and can be restarted via `restartCollectionJobs()` — the others cannot
</context>

## Task 1: Add .catch{} to permanent init collectors (M1)

**Action:** In `ActiveSessionEngine.kt`, add `.catch{}` operators to all permanent init-block collectors that lack them.

**Specific changes:**
For each of the 4 unprotected collectors, add a `.catch{}` before `.collect()`:

1. **Collector #2 (metricsFlow, ~line 145):**
   ```kotlin
   scope.launch {
       bleRepository.metricsFlow
           .catch { e -> Logger.e(e) { "metricsFlow collector error — metrics will stop" } }
           .collect { metric ->
               coordinator._currentMetric.value = metric
               handleMonitorMetric(metric)
           }
   }
   ```

2. **Collector #5 (deloadOccurredEvents, ~line 224):**
   ```kotlin
   scope.launch {
       bleRepository.deloadOccurredEvents
           .catch { e -> Logger.e(e) { "deloadEvents collector error" } }
           .collect { /* existing logic */ }
   }
   ```

3. **Collector #7 (global metricsFlow, ~line 271):**
   ```kotlin
   scope.launch {
       bleRepository.metricsFlow
           .catch { e -> Logger.e(e) { "global metricsFlow collector error" } }
           .collect { /* existing logic */ }
   }
   ```

4. **Collector #8 (heuristicData, ~line 280):**
   ```kotlin
   scope.launch {
       bleRepository.heuristicData
           .catch { e -> Logger.e(e) { "heuristicData collector error" } }
           .collect { /* existing logic */ }
   }
   ```

**Important:** The `.catch{}` operator terminates the upstream flow on exception but does NOT restart collection. For critical collectors (#2 metricsFlow), consider whether a restart mechanism is needed. For v0.8.0 beta, logging the error and silently degrading is acceptable — the collector dying is better than the app crashing.

**Also check:** Collectors #1 (connection error), #3 (rep events), #4 (handle state) — verify they already have adequate error handling or add `.catch{}` to them as well.

**Verify:** Build succeeds. Review each collector to confirm `.catch{}` is placed BEFORE `.collect()` in the chain (placement matters — `.catch{}` only catches upstream errors).

**Done when:** All permanent init-block collectors have `.catch{}` operators that log errors instead of propagating them as uncaught exceptions.
