<objective>
Deploy the `mobile-sync-pull` Supabase Edge Function that receives a pull request with a `lastSync` timestamp, verifies the GoTrue JWT, and returns all portal data modified since the last sync for the authenticated user. The response includes workout sessions (with nested exercises/sets/rep_summaries), routines, RPG attributes, badges, and gamification stats.

Purpose: This is the read path of bidirectional sync. Mobile app pulls new/updated data from the portal so the local SQLDelight database stays current with portal changes.

Output: One Edge Function file that handles the complete pull sync flow.

**Section 3: JWT verification**

Identical to Plan 01:
```typescript
const authHeader = req.headers.get('Authorization');
if (!authHeader?.startsWith('Bearer ')) {
  return new Response(JSON.stringify({ error: 'Missing or invalid authorization header' }),
    { status: 401, headers: { ...cors, 'Content-Type': 'application/json' } });
}
const token = authHeader.slice(7);

const supabaseAuth = createClient(
  Deno.env.get('SUPABASE_URL')!,
  Deno.env.get('SUPABASE_ANON_KEY')!,
  { global: { headers: { Authorization: `Bearer ${token}` } } }
);
const { data: { user }, error: authError } = await supabaseAuth.auth.getUser();
if (authError || !user) {
  return new Response(JSON.stringify({ error: 'Invalid or expired token' }),
    { status: 401, headers: { ...cors, 'Content-Type': 'application/json' } });
}
const userId = user.id;
```

**Section 11: Handle special case — lastSync = 0 (first sync)**

When lastSync is 0 (first sync / fresh install), the timestamp `new Date(0).toISOString()` = `"1970-01-01T00:00:00.000Z"`, which means ALL records will be returned. This is correct behavior — on first sync, pull everything.

For routines, they're always returned in full regardless of lastSync.

For gamification (rpg_attributes, gamification_stats): if `lastSyncISO` is `"1970-01-01..."`, the `gt('updated_at', lastSyncISO)` query will return the row since `updated_at` is always after 1970. Correct behavior.

No special handling needed — the default logic handles first sync correctly.
