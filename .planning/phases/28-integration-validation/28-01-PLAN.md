<objective>
Write comprehensive unit tests for the push/pull data transformation layer to verify data integrity across the sync boundary.

Purpose: The sync adapter layer transforms mobile domain models into portal DTOs (push) and portal DTOs back into mobile merge DTOs (pull). Bugs here cause data corruption that's hard to detect — wrong units, missing fields, incorrect grouping. These tests catch transformation errors before they reach production.

Output: 3 test files covering all adapter/mapping logic with ~40-50 test cases total.
</objective>

## Step 1: Create PortalMappingsTest.kt

Create `shared/src/commonTest/kotlin/com/devil/phoenixproject/data/sync/PortalMappingsTest.kt`

Tests (~12 cases):
- `velocityMmSToMps converts correctly` — 1000 mm/s → 1.0 m/s, 0 → 0, 500 → 0.5
- `loadKgToNewtons converts correctly` — 1.0 kg → 9.80665 N, 0 → 0, 100 → 980.665
- `newtonsToLoadKg is inverse of loadKgToNewtons` — round-trip property
- `cableToPortal maps A to left and B to right` — "A"→"left", "B"→"right", "a"→"left"
- `workoutModeToSync maps all 6 modes` — "Old School"→"OLD_SCHOOL", "Pump"→"PUMP", "TUT"→"TUT", "TUT Beast"→"TUT_BEAST", "Eccentric Only"→"ECCENTRIC_ONLY", "Echo"→"ECHO"
- `workoutModeFromSync maps all 6 sync strings` — reverse of above
- `workoutModeToSync falls back for unknown mode` — "CustomMode"→"CUSTOMMODE"
- `toPortalCategory maps granular to broad groups` — "Biceps"→"Arms", "Quads"→"Legs", "Abs"→"Core"
- `toPortalCategory passes through unknown groups` — "Custom"→"Custom"

## Step 3: Create PortalSyncAdapterTest.kt

Create `shared/src/commonTest/kotlin/com/devil/phoenixproject/data/sync/PortalSyncAdapterTest.kt`

Tests (~20 cases):

**Session grouping:**
- `standalone session produces one portal workout` — null routineSessionId → 1 workout
- `two standalone sessions produce two portal workouts` — each gets own workout
- `sessions with same routineSessionId group into one workout` — 3 sessions with same ID → 1 workout with 3 exercises
- `mixed standalone and routine sessions handled correctly` — 2 standalone + 2 grouped → 3 workouts
- `grouped workout uses routineSessionId as portal session id`
- `standalone workout uses session id as portal session id`

**Aggregation:**
- `grouped workout aggregates duration across exercises` — 30000ms + 45000ms = 75 seconds (ms→s)
- `grouped workout aggregates total volume`
- `grouped workout setCount equals number of sessions`
- `grouped workout exerciseCount equals number of sessions`

**Exercise mapping:**
- `exercise gets sequential orderIndex`
- `exercise name from session exerciseName, falls back to Unknown Exercise`
- `set weightKg from session weightPerCableKg`
- `set actualReps from session totalReps`
- `set isPr from SessionWithReps isPr flag`

**Rep summary mapping:**
- `rep summaries convert velocity mm/s to m/s`
- `rep summaries convert force kg to Newtons`
- `rep summaries compute tutMs as concentric + eccentric duration`
- `rep summaries map cable A force to leftForceAvg and cable B to rightForceAvg`

**Routine mapping:**
- `toPortalRoutine maps basic fields`
- `toPortalRoutine maps superset color names`
- `toPortalRoutine serializes per-set weights as JSON array`
- `toPortalRoutine maps Echo mode eccentricLoad and echoLevel`
- `estimateRoutineDuration returns seconds`

**Utility tests:**
- `workoutMode uses most common mode in group`
- `startedAt is ISO 8601 string of first session timestamp`
