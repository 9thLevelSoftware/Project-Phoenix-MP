```markdown
# v0.6.0 Portal Sync Compatibility — Deployment Runbook

## Overview
Deploys bidirectional sync between the Phoenix mobile app and the Phoenix Portal.
Three stages must execute in order. Each stage has verification before proceeding.

## Prerequisites
- [ ] Supabase CLI installed and linked to production project
- [ ] Portal repo (phoenix-portal) checked out at latest main
- [ ] Mobile repo (Project-Phoenix-MP) checked out at latest main
- [ ] Supabase project credentials available (project ref, service role key)
- [ ] Google Play / App Store release pipeline configured

## Stage 1: Portal Database Schema (phoenix-portal)

### Deploy
```bash
cd phoenix-portal
supabase link --project-ref <PROJECT_REF>
supabase db push
```

### Verify
1. Open Supabase Dashboard → Table Editor
2. Confirm these tables exist with correct columns:
   - `rpg_attributes` (strength, power, stamina, consistency, mastery, character_class, level, experience_points)
   - `earned_badges` (badge_id, badge_name, badge_description, badge_tier, earned_at)
   - `gamification_stats` (total_workouts, total_reps, total_volume_kg, longest_streak, current_streak, total_time_seconds)
3. Confirm `routine_exercises` has 12 advanced columns (superset_id through echo_level)
4. Confirm `exercises`, `sets`, `rep_summaries` have INSERT RLS policies
5. Run SQL: `SELECT workout_mode FROM workout_sessions LIMIT 5` — verify SCREAMING_SNAKE format

### Rollback
Schema changes are additive (new columns, new tables). No rollback needed unless migration fails:
```bash
# If migration fails, check error output and fix the SQL file
# Migrations are idempotent — re-running is safe
supabase db push
```

## Stage 2: Edge Functions (phoenix-portal)

### Deploy
```bash
supabase functions deploy mobile-sync-push
supabase functions deploy mobile-sync-pull
```

### Verify
Test push (replace TOKEN with a valid GoTrue access token):
```bash
curl -X POST https://<PROJECT_REF>.supabase.co/functions/v1/mobile-sync-push \
  -H "Authorization: Bearer TOKEN" \
  -H "apikey: ANON_KEY" \
  -H "Content-Type: application/json" \
  -d '{"deviceId":"test","platform":"android","lastSync":0,"sessions":[],"routines":[],"badges":[]}'
```
Expected: HTTP 200 with `{"syncTime":"...","sessionsInserted":0,...}`

Test pull:
```bash
curl -X POST https://<PROJECT_REF>.supabase.co/functions/v1/mobile-sync-pull \
  -H "Authorization: Bearer TOKEN" \
  -H "apikey: ANON_KEY" \
  -H "Content-Type: application/json" \
  -d '{"deviceId":"test","lastSync":0}'
```
Expected: HTTP 200 with `{"syncTime":...,"sessions":[],"routines":[],...}`

Test without Origin header (mobile client simulation):
Both commands above have no Origin header — verify they return 200 (not CORS error).

### Rollback
```bash
# Revert to previous function version
supabase functions deploy mobile-sync-push --version <PREVIOUS_VERSION>
supabase functions deploy mobile-sync-pull --version <PREVIOUS_VERSION>
```

## Stage 3: Mobile App Release (Project-Phoenix-MP)

### Pre-release Checks
- [ ] `./gradlew :androidApp:assembleRelease` succeeds
- [ ] All tests pass: `./gradlew :shared:allTests`
- [ ] Version bumped in build.gradle.kts
- [ ] Changelog updated

### Deploy
1. Build release APK/AAB
2. Upload to Google Play Console (internal testing → closed beta → production)
3. For iOS: Archive in Xcode, upload to App Store Connect

### Verify
1. Install release build on test device
2. Sign in with existing portal account
3. Complete a workout → verify it appears in portal dashboard within 30 seconds
4. Create a routine in portal → pull sync on mobile → verify routine appears
5. Force-kill app → reopen → verify session restored without login prompt

### Rollback
- Android: Halt staged rollout in Google Play Console
- iOS: Remove from sale in App Store Connect
- Edge Functions continue working — they're backwards compatible

## Post-Deployment Monitoring
- Monitor Supabase Edge Function logs for errors (Dashboard → Edge Functions → Logs)
- Monitor mobile crash reports (Firebase Crashlytics / Sentry)
- Check sync success rate: query `workout_sessions` table for recent entries from mobile users
