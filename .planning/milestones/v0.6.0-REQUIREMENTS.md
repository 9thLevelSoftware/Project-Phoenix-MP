## v0.6.0 Requirements (SHIPPED)

Requirements for v0.6.0 Portal Sync Compatibility. Each maps to roadmap phases.

**Cross-repo scope:** Mobile (Project-Phoenix-MP) + Portal (phoenix-portal). Each requirement notes target repo.

### Authentication

- [x] **AUTH-01**: User can sign in to mobile app via Supabase GoTrue REST API (email/password), receiving a valid access token that works with portal Edge Functions *(mobile)*
- [x] **AUTH-02**: Mobile app automatically refreshes expired GoTrue tokens and persists sessions across app restarts *(mobile)*
- [x] **AUTH-03**: User can register a new account from mobile app via GoTrue /auth/v1/signup, creating a single identity shared with portal *(mobile)*

### Push Sync

- [x] **PUSH-01**: SyncManager.pushLocalChanges() uses PortalSyncAdapter to build hierarchical PortalSyncPayload instead of legacy flat SyncPushRequest *(mobile)*
- [x] **PUSH-02**: Push sync includes user_id on all nested DTOs (PortalSetDto, PortalRepSummaryDto, PortalRepTelemetryDto) or Edge Function injects it server-side *(mobile + portal)*
- [x] **PUSH-03**: Push sync includes exercise_progress records computed from session/rep data (max_weight_kg, total_volume_kg, estimated_1rm_kg, max_reps, set_count) *(mobile)*
- [x] **PUSH-04**: Push sync includes personal_records matching portal schema (record_type, value, unit, achieved_at, previous_value) *(mobile)*
- [x] **PUSH-05**: Rep telemetry is explicitly excluded from the main sync payload; deferred to v0.6.1 chunked upload path. PortalSyncPayload contains no telemetry field. *(mobile)*

### Pull Sync

- [x] **PULL-01**: Mobile app can pull portal-native workout data via sync-pull Edge Function and deserialize using portal-format DTOs *(mobile + portal)*
- [x] **PULL-02**: PortalPullAdapter converts portal-native DTOs back to mobile domain models (WorkoutSession, PersonalRecord, Routine, etc.) *(mobile)*
- [x] **PULL-03**: Pull sync uses timestamp-based merge strategy: sessions immutable (push-only), routines LWW by updatedAt, badges union merge *(mobile)*

### Edge Functions

- [x] **EDGE-01**: mobile-sync-push Edge Function receives PortalSyncPayload, validates auth, and writes to all portal tables (sessions, exercises, sets, rep_summaries, rep_telemetry, exercise_progress, personal_records, routines, gamification) *(portal)*
- [x] **EDGE-02**: mobile-sync-pull Edge Function queries portal data modified since a given timestamp and returns portal-native DTOs *(portal)*
- [x] **EDGE-03**: Edge Functions use mobile-aware CORS helper that works without Origin header (native mobile clients) *(portal)*

### Portal Schema

- [x] **PORTAL-01**: rpg_attributes, earned_badges, and gamification_stats tables exist with proper RLS policies *(portal)*
- [x] **PORTAL-02**: routine_exercises table includes advanced columns (superset_id, superset_color, superset_order, per_set_weights, per_set_rest, is_amrap, pr_percentage, rep_count_timing, stop_at_position, stall_detection, eccentric_load, echo_level) *(portal)*
- [x] **PORTAL-03**: INSERT WITH CHECK RLS policies exist on exercises, sets, rep_summaries, and rep_telemetry tables *(portal)*
- [x] **PORTAL-04**: Portal frontend displays human-readable workout mode names from SCREAMING_SNAKE format *(portal)*

### Fixes

- [x] **FIX-01**: PortalSyncAdapter.estimateRoutineDuration() returns correct unit matching portal's routines.estimated_duration column expectation *(mobile)*

## Out of Scope

Explicitly excluded. Documented to prevent scope creep.

| Feature | Reason |
|---------|--------|
| Supabase Realtime subscriptions | Batch push/pull sufficient for workout sync; realtime adds complexity without clear benefit |
| supabase-kt SDK | Raw Ktor HTTP calls sufficient; SDK would create dependency conflicts with existing Ktor setup |
| Railway backend service | Edge Functions are zero-ops and match portal's existing architecture |
| Portal community features sync | Community sharing/comments/votes are portal-only features |
| Training cycles sync | Portal-only feature, future mobile feature candidate |
| External integrations sync (Strava/Fitbit/Garmin) | Portal-only; third-party OAuth handled entirely by portal |
| GraphQL layer | Portal uses REST/PostgREST; no benefit to adding query layer |
| WebSocket real-time sync | Over-engineered for batch workout sync |
| Server-side subscription validation | Client-side FeatureGate accepted for pre-launch |
| Supabase database webhooks | Not needed for mobile-initiated sync |
