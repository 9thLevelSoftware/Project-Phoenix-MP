# v0.6.0 Portal Sync Compatibility Deployment Runbook

## Overview

Project Phoenix mobile sync depends on portal schema, Supabase Edge Functions,
and the mobile release landing in the correct order. Do not publish the mobile
app until the portal database and functions are deployed and verified.

Deployment order:

1. Portal schema migrations
2. Portal Edge Functions
3. Mobile app release

Why this order matters:

- Edge Functions read and write columns created by the schema migrations.
- The mobile app calls the Edge Functions directly.
- Rolling the mobile app out first creates user-visible sync failures.

## Prerequisites

- `phoenix-portal` checked out at the release branch or latest `main`
- `Project-Phoenix-MP` checked out at the release branch or latest `main`
- Supabase CLI installed and linked to the target project
- Supabase project URL, anon key, and service role key available to the portal
  deployment environment
- Mobile release credentials configured for Google Play and App Store Connect
- Database backup taken before migration

## Stage 1: Portal Schema

Repo: `phoenix-portal`

Deploy:

```bash
supabase db push
```

Verify:

```sql
select table_name, column_name, data_type
from information_schema.columns
where table_name in (
  'rpg_attributes',
  'earned_badges',
  'gamification_stats',
  'routine_exercises'
)
order by table_name, ordinal_position;
```

Also verify:

- Sync support tables exist.
- `routine_exercises` contains the advanced routine columns used by mobile.
- Insert policies exist for portal sync write targets.
- Existing workout mode data is normalized to canonical enum values.

Do not continue to Edge Functions until schema verification passes.

## Stage 2: Edge Functions

Repo: `phoenix-portal`

Deploy:

```bash
supabase functions deploy mobile-sync-push
supabase functions deploy mobile-sync-pull
```

Verify with a real Supabase Auth access token:

```bash
curl -X POST "https://<PROJECT_REF>.supabase.co/functions/v1/mobile-sync-push" \
  -H "Authorization: Bearer <ACCESS_TOKEN>" \
  -H "apikey: <ANON_KEY>" \
  -H "Content-Type: application/json" \
  -d '{"deviceId":"deploy-test","platform":"android","lastSync":0,"sessions":[],"routines":[],"badges":[],"rpgAttributes":null,"gamificationStats":null}'
```

```bash
curl -X POST "https://<PROJECT_REF>.supabase.co/functions/v1/mobile-sync-pull" \
  -H "Authorization: Bearer <ACCESS_TOKEN>" \
  -H "apikey: <ANON_KEY>" \
  -H "Content-Type: application/json" \
  -d '{"deviceId":"deploy-test","lastSync":0}'
```

Expected results:

- Both requests return HTTP 200.
- Empty push reports zero inserted rows without error.
- Empty pull returns a valid payload with `syncTime`.
- Requests without a bearer token return HTTP 401.
- Requests without an `Origin` header still work, matching native mobile clients.

Do not continue to the mobile release until both functions are deployed and
verified.

## Stage 3: Mobile Release

Repo: `Project-Phoenix-MP`

Pre-release checks:

- Edge Function verification is complete.
- Android release build succeeds.
- iOS archive has Supabase config supplied locally or by CI secrets.
- Tracked-secret guards pass.
- No service role key is present in mobile code or mobile config.

Build Android:

```bash
./gradlew :androidApp:bundleRelease
```

Build iOS:

1. Generate or build the shared framework.
2. Open `iosApp/VitruvianPhoenix/VitruvianPhoenix.xcodeproj`.
3. Ensure `iosApp/VitruvianPhoenix/Config/Supabase.xcconfig` exists locally or
   is generated by CI.
4. Archive and upload through Xcode.

Release with a staged rollout where possible.

## Post-Release Verification

Verify on a physical device:

- Fresh sign-in succeeds.
- A completed workout pushes to the portal.
- A portal routine change pulls back to mobile.
- App restart preserves login and sync state.
- Offline workout completion syncs after reconnect.

Monitor for the first 24 hours:

- Supabase Edge Function errors and timeout rates
- Mobile crash reports
- Auth refresh failures
- Duplicate session rows by mobile client ID

## Rollback

Portal schema changes should be additive. If a migration fails, fix the failing
migration and rerun `supabase db push`; do not manually edit production data
unless there is a tested rollback script.

Edge Functions can be redeployed independently. If a function regression is
found, redeploy the previous known-good function version before pausing mobile
rollout.

For mobile regressions, halt staged rollout in the store console and ship a
patched mobile build. The Edge Functions should remain backwards compatible with
the previous mobile version.
